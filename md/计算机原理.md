# [# Your First Program](https://pwn.college/computing-101/your-first-program/)

### 寄存器 register

##### **`rax`：那个“发号施令”和“拿结果”的人**

* **身份**：系统调用号 / 返回值。
* **作用 1（输入时）**：你把一个数字放进 `rax`，告诉操作系统**“我要做什么”**。
  * 比如 `mov rax, 60`：意思是“我要执行 **退出 (exit)** 操作”。
  * 比如 `mov rax, 1`：意思是“我要执行 **写入 (write)** 操作”。
* **作用 2（输出时）**：当操作做完了，操作系统会把**结果**放在 `rax` 里还给你。
  * 比如 `write` 成功写入了 5 个字节，`rax` 就会变成 5。

##### **`rdi`：第一个参数（Argument 1）**

* **身份**：参数 1。
* **作用**：告诉操作系统执行命令时的**第一个条件**。
  * 如果是 `exit` (退出)，`rdi` 存放**退出状态码**。
  * `mov rax, 60` (我要退出) -> `mov rdi, 0` (状态码是 0，表示成功)。即 `exit(0)`。

##### **`rsi`：第二个参数（Argument 2）**

* **身份**：参数 2。
* **作用**：告诉操作系统执行命令时的**第二个条件**。
  * 如果是 `write` (写入)，`rdi` 是“写到哪去”，`rsi` 就是“**要写的内容在哪里**（内存地址）”。

##### `syscall`

表示当前命令以结束，安全从内存移出

# [# Software Introspection](https://pwn.college/computing-101/introspecting/)

##### `gdb`

进去分析：

`gdb /path/to/binary/file`

#### 1. 启动与控制

* **`starti`** (Start Instruction)：**最常用！** 程序启动并在**第一条指令**处立刻暂停。
* **`run`** (或 `r`)：直接让程序跑起来，直到遇到断点（比如你写的 `int3`）或程序结束。
* **`c`** (Continue)：继续运行，直到下一个断点。
* **`q`** (Quit)：退出 GDB。

#### 2. 单步执行 (微操)

* **`si`** (Step Instruction)：**非常重要！** 执行**一条**汇编指令，然后停下来。
  * *注：C语言调试用 `s` (step)，汇编调试必须用 `si`。*
* **`ni`** (Next Instruction)：执行一条指令，但如果是函数调用（call），它不会进函数内部，而是直接把函数执行完。

#### 3. 查看状态 (偷看)

* **`i r`** (Info Registers)：**查看所有寄存器的值**。
  * 想看 `rax` 是多少？输这个就对了。
* **`x`** (Examine Memory)：**查看内存**。
  * `x/s 0x402000`：把这个地址的内容当做**字符串**看。
  * `x/i $rip`：查看即将执行的指令是什么。
  * `x/10gx $rsp`：查看栈顶的10个数值。
  * 
  * **`x/a`**：用来**获取指针**（看地址指向哪里）。
  * **`x/s`**：用来**读取内容**（看那个地址里存了什么字）。
* **`disas`** (Disassemble)：查看当前函数的汇编代码。

#### 4. 断点

* **`b *0x401000`** (Break)：在地址 `0x401000` 处设置断点。
* **`int3`**：你在代码里写的 `int3` 指令，GDB 会自动把它当做断点处理。

# [# Output and Input](https://pwn.college/computing-101/hello-hackers/)

##### 文件描述符(FD)

- FD 0：标准输入是进程接收输入的通道。例如，你的 shell 使用标准输入来读取你输入的命令。
- FD 1：标准输出是进程输出正常数据的通道，例如在之前的挑战中打印给您的标志，或者诸如之类的实用程序的输出`ls`。
- FD 2：标准错误输出是进程输出错误详情的通道。例如，如果您输入错误的命令，shell 会通过标准错误输出提示该命令不存在。

##### `write(file_descriptor, memory_address, number_of_characters_to_write)`

```
mov rdi, 1 #fd1
mov rsi, memory_address

```

```

```

##### 关于`syscall`

并不是每条指令都要 `syscall`。

* **不需要 `syscall`**：
  
  * `mov` (搬运数据)
  * `add`, `sub` (算术运算)
  * `cmp`, `jmp` (逻辑判断跳转)
  * **原因**：这些是 CPU 自己就能干的活，不需要麻烦操作系统。

* **必须 `syscall`**（涉及外部资源）：
  
  * **文件操作** (`open`, `read`, `write`, `close`)
  * **进程控制** (`exit`, `fork`, `execve`)
  * **内存管理** (`mmap`，申请新内存)
  * **网络通信** (`socket`, `connect`)

##### 关于 `rax` 与 `rdi`

| 数字    | 放在 `rax` 里 (动词)   | 放在 `rdi` 里 (我是/参数)    |
|:----- |:----------------- |:--------------------- |
| **0** | **Read** (读取函数)   | **Stdin** (键盘/输入源)    |
| **1** | **Write** (写入函数)  | **Stdout** (屏幕/输出目的地) |
| **2** | **Open** (打开文件函数) | **Stderr** (屏幕/错误输出)  |

##### 关于 `rsi`

简单一句话总结：**`rsi` 永远是系统调用的“第二个参数”**。

但是，根据 `rax`（你要做的动作）不同，这个“第二个参数”的具体含义会有很大的区别。我们来看看你遇到过的三种情况：

###### 1. 在 `write` (写) 和 `read` (读) 中

**`rsi` 的含义：数据的内存地址（Buffer Pointer）。**

在这两个最常用的功能里，`rsi` 回答了操作系统的一个核心问题：**“数据在哪？”**

* **当 `rax = 1` (Write / 写) 时**：
  
  * **含义**：`rsi` 指向**数据源**。
  * **潜台词**：“操作系统，请去 **`rsi` 指向的内存地址** 拿数据，把那里的东西打印到屏幕上。”
  * *例子*：`mov rsi, rsp`（数据在栈顶，去栈顶拿）。

* **当 `rax = 0` (Read / 读) 时**：
  
  * **含义**：`rsi` 指向**目的地**。
  * **潜台词**：“操作系统，请把读进来的数据，存放到 **`rsi` 指向的内存地址** 里去。”
  * *例子*：`mov rsi, rsp`（把键盘敲进来的字存在栈顶）。

> **关键点**：在这两种情况下，`rsi` 里存的必须是一个**有效的内存地址**（指针）。如果你存了 `1` 或者 `0`，操作系统找不到这个地址，就会报错（Bad Address）。

###### 2. 在 `open` (打开文件) 中

**`rsi` 的含义：打开模式（Flags / Mode）。**

这里不一样了！因为打开文件不需要“数据缓冲区”，所以这里的第二个参数变成了“选项”。

* **当 `rax = 2` (Open / 打开) 时**：
  * **含义**：`rsi` 代表**怎么打开文件**。
  * **潜台词**：“操作系统，我要打开这个文件，请用 **`rsi` 指定的模式**。”
  * **常用值**：
    * `mov rsi, 0`：**只读模式** (O_RDONLY)。
    * `mov rsi, 1`：**只写模式** (O_WRONLY)。
    * `mov rsi, 2`：**读写模式** (O_RDWR)。

> **关键点**：在这种情况下，`rsi` 里存的是一个**纯数字**（配置开关），而不是内存地址。

`BYTE PTR` 是汇编语言中用来**指定内存操作大小**的一个修饰符。

简单来说，它的意思是：**“我只操作这个地址处的 1 个字节（8 bit）。”**

为了让你彻底理解，我们需要从“歧义”说起。

##### 关于 `BYTE PTR`

###### 1. 为什么要加 `BYTE PTR`？（消除歧义）

想象一下，内存就像是一长排的**小格子**，每个格子能装 1 个字节（Byte）。

如果你写了这样一条指令：

```assembly
mov [rsp], 1
```

* **`[rsp]`**：告诉 CPU 去 `rsp` 指向的那个地址（比如第 100 号格子）。
* **`1`**：告诉 CPU 把数字 1 放进去。

**这时候 CPU 和汇编器就懵了**，它们会问你一个问题：

> “大哥，数字 1 是很小，但你希望我占几个格子？”

* 是占 **1 个格子**？（`01`）
* 是占 **2 个格子**？（`00 01`）
* 还是占 **4 个格子**？（`00 00 00 01`）
* 还是占 **8 个格子**？（`00 00 00 00 00 00 00 01`）

因为 `[rsp]` 只是一个地址，没有包含大小信息；而立即数 `1` 也没有大小信息。**这就是歧义。**

为了消除歧义，你必须显式地告诉 CPU：

```assembly
mov BYTE PTR [rsp], 1
```

这就清楚了：**“去 `rsp` 指向的地址，只用 1 个格子，把 1 放进去。”**

###### 2. 常见的 PTR 家族

根据你要操作的数据大小不同，有以下几种写法（在 x86-64 中最常用）：

| 写法              | 全称          | 大小  | 字节数         | 对应的寄存器例子            |
|:--------------- |:----------- |:--- |:----------- |:------------------- |
| **`BYTE PTR`**  | Byte        | 字节  | **1** Byte  | `al`, `bl`, `dil`   |
| **`WORD PTR`**  | Word        | 字   | **2** Bytes | `ax`, `bx`, `di`    |
| **`DWORD PTR`** | Double Word | 双字  | **4** Bytes | `eax`, `ebx`, `edi` |
| **`QWORD PTR`** | Quad Word   | 四字  | **8** Bytes | `rax`, `rbx`, `rdi` |

###### 3. 在你刚才的代码中

你写了：

```assembly
cmp BYTE PTR [rsp], 42
```

这句话的意思是：

1. 去 `rsp` 指向的内存地址。
2. **只读取这 1 个字节的数据**。
3. 把这 1 个字节的数据拿来和 `42` 做比较。
4. 忽略该地址后面跟着的其他数据。

**为什么要这么写？**

* 因为 `argc`（参数数量）通常是一个很小的数字（比如 1, 2, 3...）。
* 虽然 `argc` 在栈上实际上占了 8 个字节（64位系统栈对齐），但我们只需要看它的**最低位**那个字节就足够判断它是不是 42 了。
* 如果我们写 `cmp QWORD PTR [rsp], 42`，也是可以的，那样 CPU 就会读完整的 8 个字节来比较。但在这一题里，题目特意引导你用 `BYTE PTR` 来练习对内存的精细控制。

###### 总结

* **`PTR`** = 指针（Pointer），表示后面跟的是个内存地址。
* **`BYTE`** = 大小，表示这次操作只涉及 1 个字节。
* **什么时候用？** 当你操作内存 `[]`，且另一边的操作数是常数（立即数），导致 CPU 猜不出大小时，必须加这个前缀。

### `strings`

strings 命令会遍历整个文件（哪怕它是个图片、压缩包或二进制程序），找出其中**连续的、可打印的 ASCII 字符**（默认长度 >=4），并把它们按行打印出来。

它**完全不关心**文件的内部结构（比如它不管哪里是代码区，哪里是数据区）。它只做无脑的模式匹配：只要连续遇到几个能在屏幕上打出来的字母数字，并且以 `\0`（空字符/结束符）结尾，它就把这一串字提取出来。

* **基础扫描**：`strings ./program` (直接输出所有字符串)
* **配合 `grep` 过滤**：`strings ./program | grep "flag"` (只找包含 flag 关键字的句子)
* **配合 `less` 分屏**：`strings ./program | less` (方便上下翻页查看)
* **🌟 `-n` 指定最短长度**：`strings -n 6 ./program` (只显示长度大于等于 6 的字符串，能**过滤掉大量无用的两三个字母的垃圾信息**)
* **🌟 `-t x` 显示字符串的地址**：`strings -t x ./program` (在字符串前面显示它在文件中的十六进制偏移地址，方便你后续用 GDB 精准定位)

**优点**：极快、极简。是拿到任何未知文件后**第一步必敲的命令**（用来找硬编码的密码、报错提示、隐藏的网址等）。

**缺点**：噪音太大（会把很多无关的系统函数名也打出来）；如果字符串被加密或动态生成，它就什么也找不到。

### `objdump`

1. **它是干什么的？**

`objdump`（Object Dump）是用来展示二进制文件（在 Linux 中通常是 ELF 格式）详细信息的工具。它**看得懂二进制文件的内部结构**，知道哪里是代码、哪里是数据。

2. **程序的结构（背景知识）**

Linux 程序在内部是被划分为多个**区段（Sections）**的，最常见的三个：

* **`.text`**：代码区（存放机器码，即 `mov`, `add`, `syscall` 这些指令）。
* **`.rodata`**：只读数据区 (Read-Only Data)（存放程序里写死的常量，比如 `"Hello World"`、密码等）。
* **`.data` / `.bss`**：读写数据区（存放全局变量）。
3. **常用的实战命令**

`objdump` 的功能极多，靠不同的参数来切换“解剖模式”：

 模式 A：看代码逻辑（反汇编）

* **命令**：`objdump -d -M intel ./program`
* **参数解释**：
  * `-d` (disassemble)：反汇编，将机器码翻译成汇编代码（主要针对 `.text` 区段）。
  * `-M intel`：使用易读的 Intel 语法（目标在左，源在右，不用 `%` 和 `$` 符号），代替默认晦涩的 AT&T 语法。
* **用途**：看程序的控制流，找 `cmp` 和 `jmp` 看它是怎么判断密码的。

 模式 B：看静态数据（查表/查常量）

* **命令**：`objdump -s -j .rodata ./program`
* **参数解释**：
  * `-s` (full contents)：显示指定区段的完整十六进制内容（Hex Dump）。
  * `-j .rodata` (section)：只提取 `.rodata` 这一块区域的内容。
* **用途**：精准查看程序硬编码的字符串、跳转表（Jump Table）数组。

模式 C：看程序大纲

* **命令**：`objdump -h ./program`
* **参数解释**：`-h` (headers)，显示文件的各个区段头部信息（可以看到这个程序有哪些区段，每个区段占用多大空间）。
4. **优缺点**
* **优点**：专业、精准。不会像 `strings` 那样混入无用的垃圾信息。能还原完整的逻辑。
* **缺点**：输出内容需要具备一定的汇编语言和内存布局知识才能看懂；输出有时过长，需要配合 `| less` 或 `| grep` 阅读。
